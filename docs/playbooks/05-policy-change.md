# 05 — Policy change

> **Sensitive surface.** Any edit under `policies/` changes the
> deterministic guardrails enforced on every PR. A silent gap here
> means unsafe Terraform or Bicep ships. **Never agent-eligible.**
> Failing fixture before passing rule, in the same PR.

## Triggers

- Diff includes any `*.rego` file under `policies/`.
- Diff includes any conftest fixture under `orchestrator/tests/data/`
  intended to exercise a policy rule.

## Pre-flight: this is human-only

If a cloud agent reached this playbook, stop and unassign — the squad
bootstrap should not have routed you here. Comment with the
sensitive-path one-liner.

## Steps (human-author only)

1. **Failing fixture first.** Add the conftest fixture (typically a
   `*.json` plan or a `*.tf` snippet) that the rule should reject —
   *before* writing the rule. Run `conftest test` and confirm it fails
   with the expected denial reason. If it passes, the fixture is wrong.
2. **Write the rule.** One `deny` rule per concern, one message per
   denial. Reuse the existing helpers (`allowlist_has`, `regex.match`)
   to keep ruleset style consistent with `policies/avm_pinning.rego`.
3. **Add a passing fixture.** Add at least one fixture that the rule
   *does not* reject. Together the failing + passing fixtures pin the
   rule's boundary.
4. **Document the package.** The rule's package comment (top of file)
   must describe the input shape, expected denial messages, and the CI
   step that runs it. The autogenerated `docs/generated/policies.md`
   index pulls from these comments.
5. **Mind the multi-model judge.** Policy changes are in the judge
   surface (per `docs/multi-model-judge.md`). Attach
   `judge-verdicts.json` from ≥ 2 providers.
6. **Defence in depth.** If a sibling schema can express the same
   constraint, add it there too — schemas reject malformed input
   before OPA sees it. See `schemas/rcg.schema.json` (which rejects
   wildcards even though `policies/firewall_rules.rego` also does).

## Definition of Done

- [ ] A failing-fixture commit precedes the policy commit (or both
      land in a single squashed commit accompanied by both fixtures).
- [ ] `ci / OPA policies (rego syntax + unit)` passes.
- [ ] `ci / orchestrator (lint + test)` passes (Python tests that
      exercise policy outputs are unaffected or updated).
- [ ] `ci / lint-instructions` passes.
- [ ] `docs / generate-and-check` passes (regenerated `policies.md`).
- [ ] `judge-verdicts.json` from ≥ 2 distinct providers attached to
      the PR (per `docs/multi-model-judge.md`).
- [ ] `rubberduck / check` passes.
- [ ] NetSec / platform CODEOWNER approved.
