# 10 — Research and decide

> **Mandatory for any architectural decision touching a sensitive
> surface.** This is the formalisation of the "research → N MDs →
> aggregate → rubberduck" loop. Output is a typed `decision/<id>/`
> folder, a unanimous-PASS `judge-verdicts.json`, and an ADR.

## Triggers

- A PR governed by `04-prompt-or-schema-change.md`,
  `05-policy-change.md`, `06-iac-template-change.md`, or
  `08-ci-workflow-change.md` requires an architectural choice that is
  not obvious from the existing code.
- You are about to choose an architectural option based on one model's
  output. Single-model "consensus" is refused by code
  (`agentic_alz/llm/judge.py`); use this playbook instead.

## Steps

1. **Open `decision/<id>/` and write `problem.md`.** State the
   problem, the worst-case if we get it wrong, the mutually-exclusive
   options (≥ 2), and the criteria the rubric will score (re-use the
   `docs/multi-model-judge.md` rubric where it fits; add criteria
   sparingly).
2. **Deep research, read-only.** Use `web` and the `microsoft-learn-mcp`
   server in `read` mode (per `docs/mcp.allowlist.yaml`). Capture the
   primary sources, with retrieval timestamps, in `references.md`. No
   write-mode MCP calls in this playbook.
3. **One markdown per allow-listed frontier model.** For each model
   eligible to play the `judge` role in `docs/models.allowlist.yaml`,
   send the rubric + problem + references and write the model's
   verdict to `decision/<id>/<provider>-<model>.md`. Include ≥ 2
   distinct providers (enforced by `agentic_alz/llm/judge.py`).
4. **Aggregate.** Run:

   ```bash
   agentic-alz judge \
     --rubric decision/<id>/rubric.json \
     --inputs 'decision/<id>/*.md' \
     --out   decision/<id>/judge-verdicts.json
   ```

   The aggregator is deterministic given the inputs.
5. **Iterate until unanimous PASS.** If any criterion FAILs, do not
   override by majority. Either change the proposal or capture the
   dissent in `decision/<id>/dissent.md` and stop — the disagreement
   is escalated to a human.
6. **Synthesise `decision.json`** validated by
   `schemas/decision.schema.json`.
7. **Write the ADR.** Use `docs/adr-template.md` to produce
   `docs/adr/<NNNN>-<slug>.md`; cross-link from `decision.json`'s
   `adr.path`. The ADR is the artefact that ships in the
   implementation PR.
8. **Rubberduck.** The implementation PR's `## Rubberduck` section
   cites the decision folder in `## What I considered and rejected`
   and the ADR in `## Self-review notes`.

## What this playbook is *not*

- Not a substitute for the schema / policy / template playbooks. It
  produces the *decision*; the surface playbook produces the *diff*.
- Not a runtime gate. Apply still requires a human in a protected
  environment.
- Not a way to get an LLM to author a sensitive-surface PR. The
  outputs of this playbook are markdown and JSON; the implementation
  PR is human-authored.

## Definition of Done

- [ ] `decision/<id>/decision.json` validates against
      `schemas/decision.schema.json`.
- [ ] `decision/<id>/judge-verdicts.json` shows unanimous PASS per
      criterion (default) or documented majority opt-down.
- [ ] At least two distinct providers among the verdicts.
- [ ] An ADR under `docs/adr/` is in the same PR as the
      implementation, linked from `decision.json`.
- [ ] `ci / orchestrator (lint + test)` passes (decision-schema
      round-trip).
- [ ] `ci / lint-instructions` passes.
- [ ] `docs / generate-and-check` passes (the autogenerated
      `docs/generated/decisions-index.md` regenerates without diff).
- [ ] `rubberduck / check` passes on the implementation PR; the
      rubberduck cites this decision folder.
